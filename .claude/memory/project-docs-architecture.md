name: Dual docs architecture
description: Two separate doc systems — markdown in docs/ for AI/text, pdoc HTML via GitHub Actions for GitHub Pages; docstrings are Google style
type: project
---

The project maintains two sets of documentation with different purposes:

- `docs/` — markdown files generated by `scripts/make-api-docs.py` (run via `uv run inv docs`), committed to the repo. Intended for AI/text consumption and readable directly on GitHub. Includes `_config.yml` for Jekyll theming as fallback.
- GitHub Pages HTML — generated by pdoc via `.github/workflows/pages.yml` on every push to main, deployed automatically to `https://furechan.github.io/mintalib/`. Never committed to the repo.

Docstrings (in the cython `.pxi` files) are **Google style** (`Args:` sections, `period (int): description` — no space before the colon). Both renderers must be told: pdoc runs with `-d google` in pages.yml, and make-api-docs.py converts docstrings to markdown via `pdoc.docstrings.google()`. The Cython `embedsignature` directive was removed (July 2026) — `binding=True` provides introspection, `core.pyi` provides typed stubs; do not re-add it, it duplicates signatures in the docs.

**Why:** Markdown is better for AI tools and plain readability; pdoc HTML is more attractive for human browsing. Generating HTML in CI avoids committing build artifacts.

**How to apply:** When updating docs, run `uv run inv docs` for the markdown side. The HTML side is handled automatically by CI on push.
